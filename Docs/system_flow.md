# NSD – System Flow

## Overview

This document describes the data flow and processing flow of the NSD (Network Suspicious Detection) system.

The system collects events from devices, processes them through the detection engine, generates alerts, manages investigation cases, and stores evidence and reports.

The flow represents how data moves through the system from ingestion to investigation and reporting.

---

## High Level Flow

Device / Sensor / Log / Cloud Service
        ↓
Event Ingestion API
        ↓
Event Storage
        ↓
Detection Engine
        ↓
Risk Scoring
        ↓
Alert Engine
        ↓
Alert Management
        ↓
Case Management
        ↓
Evidence Management
        ↓
Report Generation
        ↓
Dashboard / Admin / External Systems

---

## Detailed System Flow

### 1. Event Generation

Events are generated by various sources:

- IoT devices
- GPS trackers
- Cameras
- Access control systems
- Network logs
- Authentication systems
- Cloud services
- Mobile applications
- Servers and routers

Example events:
- login attempt
- device movement
- door open
- camera motion detected
- failed authentication
- configuration change
- unusual access time
- system error
- network anomaly

These events are sent to the Event Ingestion API.

---

### 2. Event Ingestion

The Event API receives incoming events from devices and systems.

Responsibilities:
- authenticate device or system
- validate event format
- attach metadata (timestamp, organization, location)
- normalize event structure
- store raw event
- push event to detection engine

Output:
- stored event
- event forwarded to detection engine

---

### 3. Event Storage

All events are stored in the database for:
- audit trail
- investigation
- analytics
- evidence
- machine learning (future)

Event data includes:
- event_id
- device_id
- user_id
- timestamp
- location_id
- event_type
- severity
- raw_data
- metadata

---

### 4. Detection Engine

The Detection Engine analyzes events and behavior patterns.

Detection methods may include:
- rule-based detection
- behavior anomaly detection
- frequency analysis
- time-of-day anomaly
- geo movement anomaly
- repeated failures
- risk score accumulation
- device behavior change
- user behavior change

The detection engine produces risk signals and risk scores.

---

### 5. Risk Scoring

Each event, device, and user may receive a risk score.

Risk score sources:
- suspicious event type
- repeated failed logins
- unusual time access
- abnormal movement
- rule match
- anomaly detection
- blacklist match
- manual analyst input

Risk scores are stored and used to determine whether alerts should be generated.

---

### 6. Alert Generation

Alerts are generated when:
- risk score exceeds threshold
- detection rule triggers
- anomaly detected
- multiple suspicious events occur
- manual alert created

Alert includes:
- alert_id
- risk_score
- severity
- priority
- related device
- related user
- related events
- description
- status

Alerts are sent to operators and stored in the system.

---

### 7. Alert Management

Operators can:
- review alerts
- assign alerts
- comment on alerts
- change alert status
- escalate alert to case
- link alerts together
- mark false positive
- close alert

Alert statuses:
- open
- investigating
- escalated
- closed
- false_positive

---

### 8. Case Management

If an alert requires investigation, it becomes a Case.

Case management includes:
- case creation
- investigation notes
- evidence attachment
- linking multiple alerts
- assignment to investigator
- status tracking
- timeline tracking
- final report

Case statuses:
- open
- investigating
- monitoring
- resolved
- closed

---

### 9. Evidence Management

Evidence may include:
- logs
- images
- video
- documents
- network captures
- exported reports
- screenshots
- GPS history
- access history

Evidence must be:
- immutable
- timestamped
- auditable
- securely stored

---

### 10. Report Generation

Reports may include:
- incident reports
- investigation reports
- statistics reports
- risk reports
- activity reports
- compliance reports

Reports can be exported as:
- PDF
- CSV
- JSON
- Dashboard view

---

### 11. Dashboard / Admin / External Systems

The dashboard allows:
- alert monitoring
- case management
- device management
- user management
- analytics
- system monitoring
- rule management
- report viewing

External systems may connect via API.

---

## End-to-End Flow Summary

Full flow:

Device → Event → Detection → Risk Score → Alert → Case → Evidence → Report → Dashboard

This flow represents the core workflow of the NSD platform.
