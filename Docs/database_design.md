NSD – Database Design

Overview

This document describes the database design for NSD (Network Suspicious Detection).
The database stores users, devices, events, alerts, cases, locations, and audit logs.

The system is designed to support large-scale event ingestion, alert management, and investigation workflows.

⸻

Database Type

Recommended database:
	•	PostgreSQL (Primary relational database)
	•	Redis (Cache / Session / Temporary risk scoring)
	•	Object Storage (Logs / Images / Backups)

⸻

Main Tables

Users

Stores system users such as operators and administrators.

Column	Type	Description
user_id	UUID	Primary key
name	VARCHAR	User name
email	VARCHAR	Email
role	VARCHAR	admin / operator / viewer
organization	VARCHAR	Organization
created_at	TIMESTAMP	Creation time


⸻

Devices

Stores registered devices such as GPS trackers, cameras, sensors.

Column	Type	Description
device_id	UUID	Primary key
device_type	VARCHAR	gps / camera / sensor / mobile
owner_user_id	UUID	Device owner
status	VARCHAR	active / inactive
registered_at	TIMESTAMP	Registration time


⸻

Events

Stores incoming device events and telemetry data.

Column	Type	Description
event_id	UUID	Primary key
device_id	UUID	Device ID
timestamp	TIMESTAMP	Event time
location_lat	DECIMAL	Latitude
location_lng	DECIMAL	Longitude
event_type	VARCHAR	movement / login / tamper / offline
raw_data	JSONB	Raw event data


⸻

Alerts

Stores alerts generated by detection engine.

Column	Type	Description
alert_id	UUID	Primary key
event_id	UUID	Related event
risk_score	INT	Risk score
alert_level	VARCHAR	low / medium / high
created_at	TIMESTAMP	Created time
status	VARCHAR	open / investigating / closed


⸻

Cases

Investigation cases created from alerts.

Column	Type	Description
case_id	UUID	Primary key
alert_id	UUID	Related alert
assigned_to	UUID	Assigned operator
status	VARCHAR	open / investigating / resolved
notes	TEXT	Investigation notes
created_at	TIMESTAMP	Created time


⸻

Locations

Geofence locations and monitored areas.

Column	Type	Description
location_id	UUID	Primary key
name	VARCHAR	Location name
latitude	DECIMAL	Latitude
longitude	DECIMAL	Longitude
geo_fence_radius	INT	Radius (meters)


⸻

Risk Scores

Stores risk scoring history.

Column	Type	Description
score_id	UUID	Primary key
event_id	UUID	Related event
score	INT	Risk score
reason	TEXT	Reason for scoring
calculated_at	TIMESTAMP	Calculation time


⸻

Audit Logs

Stores system activity logs.

Column	Type	Description
log_id	UUID	Primary key
user_id	UUID	User
action	VARCHAR	Action performed
timestamp	TIMESTAMP	Time
details	TEXT	Details


⸻

Entity Relationship Overview

Relationship summary:
	•	One User → Many Devices
	•	One Device → Many Events
	•	One Event → One Alert
	•	One Alert → One Case
	•	Events → Risk Scores
	•	Users → Audit Logs
	•	Locations → Used for Geofence Detection

⸻

Index Strategy

Recommended indexes:

Table	Index
events	device_id
events	timestamp
alerts	status
alerts	risk_score
cases	assigned_to
audit_logs	user_id
locations	latitude, longitude

Indexes are important because the system will store large volumes of event data.

⸻

Data Retention Policy

Data	Retention
Events	1–3 years
Alerts	5 years
Cases	5–10 years
Audit Logs	5 years
Raw Logs	Object Storage

Old event data can be archived to cold storage.
