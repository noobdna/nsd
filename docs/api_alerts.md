NSD – Alerts API

Overview

The Alerts API manages suspicious activity alerts generated by the detection engine.

Alerts are created when events or behavior patterns exceed defined risk thresholds or match detection rules. Alerts are used by operators to review suspicious activity and decide whether to escalate to a case.

The Alerts API supports:
	•	alert creation
	•	alert listing
	•	alert details
	•	alert status updates
	•	alert assignment
	•	alert escalation to case
	•	alert comments
	•	alert audit trail
	•	alert SLA tracking
	•	alert deduplication

Alert SLA

Response Time SLA

Severity	Response Time
critical	5 minutes
high	30 minutes
medium	4 hours
low	24 hours

The system calculates:

sla_due_at = alert_created_at + SLA_time

Alert Deduplication

To prevent alert flooding, similar alerts should be merged.

Deduplication Keys

Field
rule_id
device_id
user_id
time_window

If alerts match within a time window:
	•	Update existing alert
	•	Increase event_count
	•	Do not create a new alert

Example:

{
  "alert_id": "ALT-20260330-0001",
  "event_count": 47
}

API Endpoints

Get Alerts List

GET /api/v1/alerts

Query Parameters

Parameter	Description
status	Filter by status
severity	Filter by severity
priority	Filter by priority
device_id	Filter by device
user_id	Filter by user
assigned_to	Assigned analyst
from	Start time
to	End time
limit	Number of results
offset	Pagination offset

Example

GET /api/v1/alerts?status=open&severity=high

Create Alert (Detection Engine)

POST /api/v1/alerts

Request

{
  "severity": "high",
  "priority": "P2",
  "risk_score": 82,
  "title": "Multiple failed login attempts",
  "description": "Device DEV-001 failed login 12 times within 5 minutes",
  "device_id": "DEV-001",
  "user_id": "USR-009",
  "rule_id": "RULE-LOGIN-FAIL",
  "event_count": 12
}


⸻

Update Alert Status

PATCH /api/v1/alerts/{alert_id}/status

{
  "status": "investigating"
}


⸻

Assign Alert

PATCH /api/v1/alerts/{alert_id}/assign

{
  "assigned_to": "USR-ADMIN-01"
}


⸻

Escalate Alert to Case

POST /api/v1/alerts/{alert_id}/escalate

This will:
	1.	Create a case
	2.	Link alert to case
	3.	Change alert status to escalated

Response

{
  "case_id": "CASE-20260330-0012",
  "alert_id": "ALT-20260330-0001",
  "status": "escalated"
}


⸻

Add Alert Comment

POST /api/v1/alerts/{alert_id}/comments

{
  "comment": "Investigating login logs",
  "created_by": "USR-ADMIN-01"
}

Audit Trail

All alert actions must be logged:
	•	alert created
	•	status changed
	•	assigned
	•	escalated
	•	comment added
	•	closed
	•	marked false positive

Example audit log entry:

{
  "audit_id": "AUD-000123",
  "entity": "alert",
  "entity_id": "ALT-20260330-0001",
  "action": "status_changed",
  "old_value": "open",
  "new_value": "investigating",
  "changed_by": "USR-ADMIN-01",
  "timestamp": "2026-03-30T09:20:11Z"
}
