NSD – Backend API Design

Overview

This document describes the backend API design for NSD (Network Suspicious Detection).

The backend API is responsible for receiving device data, managing alerts and cases, handling users and authentication, and providing data to the frontend dashboard.

⸻

Backend Responsibilities

The Backend API handles the following:
	•	User authentication and authorization
	•	Device registration and management
	•	Data ingestion from IoT / GPS / sensors
	•	Event storage and retrieval
	•	Risk scoring requests
	•	Alert generation and management
	•	Case management
	•	Dashboard data delivery
	•	Audit logging
	•	System configuration

⸻

API Architecture

Recommended architecture:
	•	REST API (primary)
	•	JSON request / response
	•	HTTPS only
	•	Token-based authentication (JWT)
	•	Role-Based Access Control (RBAC)
	•	Optional WebSocket for real-time alerts

⸻

Main API Modules

1. Authentication API

Handles login, logout, token management, and user sessions.

Endpoints
	•	POST /api/v1/auth/login
	•	POST /api/v1/auth/logout
	•	GET /api/v1/auth/me
	•	POST /api/v1/auth/refresh

⸻

2. User Management API

Handles user accounts and roles.

Endpoints
	•	GET /api/v1/users
	•	POST /api/v1/users
	•	GET /api/v1/users/{id}
	•	PUT /api/v1/users/{id}
	•	DELETE /api/v1/users/{id}

⸻

3. Device Management API

Handles device registration and device status.

Endpoints
	•	GET /api/v1/devices
	•	POST /api/v1/devices
	•	GET /api/v1/devices/{id}
	•	PUT /api/v1/devices/{id}
	•	DELETE /api/v1/devices/{id}
	•	GET /api/v1/devices/{id}/events

⸻

4. Event / Data Ingestion API

Receives GPS data, sensor data, and device events.

Endpoints
	•	POST /api/v1/events
	•	POST /api/v1/gps
	•	POST /api/v1/sensors
	•	GET /api/v1/events
	•	GET /api/v1/events/{id}

⸻

5. Risk Scoring API

Used by the Detection Engine to calculate risk scores.

Endpoints
	•	POST /api/v1/risk/score
	•	GET /api/v1/risk/{event_id}

⸻

6. Alert Management API

Handles alerts generated by the system.

Endpoints
	•	GET /api/v1/alerts
	•	GET /api/v1/alerts/{id}
	•	PUT /api/v1/alerts/{id}
	•	POST /api/v1/alerts/{id}/acknowledge
	•	POST /api/v1/alerts/{id}/close

⸻

7. Case Management API

Used by operators to manage investigation cases.

Endpoints
	•	GET /api/v1/cases
	•	POST /api/v1/cases
	•	GET /api/v1/cases/{id}
	•	PUT /api/v1/cases/{id}
	•	POST /api/v1/cases/{id}/assign
	•	POST /api/v1/cases/{id}/close

⸻

8. Dashboard API

Provides aggregated data for frontend dashboard.

Endpoints
	•	GET /api/v1/dashboard/summary
	•	GET /api/v1/dashboard/alerts
	•	GET /api/v1/dashboard/events
	•	GET /api/v1/dashboard/map
	•	GET /api/v1/dashboard/stats

⸻

9. Audit Log API

Tracks user actions and system changes.

Endpoints
	•	GET /api/v1/audit
	•	GET /api/v1/audit/{id}

⸻

Authentication & Authorization

Authentication
	•	JWT Token
	•	OAuth (optional future)
	•	API Key for devices

Authorization (Roles)

Example roles:
	•	Admin
	•	Operator
	•	Analyst
	•	Viewer
	•	Device

Each role has different API access permissions.

⸻

Example API Flow

Device Event Flow
	1.	Device sends GPS/event data → /events
	2.	Backend stores event
	3.	Detection Engine analyzes event
	4.	Risk score calculated
	5.	Alert generated if necessary
	6.	Alert appears on Dashboard
	7.	Operator creates Case
	8.	Case investigated and closed

⸻

Error Handling

Standard response format:

{
  "status": "error",
  "message": "Invalid device ID",
  "code": 400
}

Success response:

{
  "status": "success",
  "data": { }
}
